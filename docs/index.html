LWRT — Lightweight Wireless Router

OpenWrt's userspace, reimagined as one compact static Rust binary riding inside an execute-in-place (XIP) Linux kernel — small enough for a 4 MB / 4 MB router. The kernel keeps the hard parts (nftables, NAT, the bridge/DSA datapath, WireGuard, mt76); LWRT replaces procd, netifd, ubus, uci, dnsmasq, fw4, opkg and uhttpd with a single multicall executable.

Project Overview

LWRT is a complete router operating-system userspace for memory-constrained wireless routers such as the TP-Link Archer C54 v1 (MediaTek MT7628, 4 MB SPI flash). Instead of a sprawling stack of C daemons, shared libraries and an interpreter, the entire userspace is one statically-linked Rust binary built against musl.

A modern OpenWrt image cannot fit a 4 MB flash. LWRT buys the room back two ways: the userspace collapses to a single small executable, and the kernel runs execute-in-place directly from flash — freeing the ~2.3 MB of RAM stock Linux spends on a decompressed image. The XIP kernel lives in its own repository (mips-xip-kernel); LWRT points at it at build time and does not vendor it.

Why LWRT

A typical SoHo router — e.g. the Archer C54: MT7628, 64 MB RAM but only 4 MB SPI flash — cannot hold a modern OpenWrt image. Two ideas buy the room back.

1. An execute-in-place kernel

On MIPS, KSEG0/KSEG1 are unmapped segments, so the kernel's .text, .rodata and .init.text can run directly from the flash window with no page tables and no copy to RAM. That frees roughly 2.3 MB of RAM that stock Linux spends on a decompressed kernel image.

2. One tiny userspace binary

Instead of dozens of C daemons plus shared libraries plus an interpreter stack, LWRT ships a single statically-linked Rust binary with musl. No libubus, no Lua, no shell.

The WireGuard lesson, applied everywhere

WireGuard wins by keeping the hard work — the crypto, the transport, the routing — inside ~4k lines of kernel code, and shipping a tiny userspace tool that just pushes keys and peers in. LWRT generalises that discipline: every applet is a thin front end to an in-kernel feature. The firewall applet programs nftables over netlink; the VPN applet creates the wireguard link and pushes the device config; nobody re-implements a packet path in userspace.

Design rule. If the kernel already does it well, LWRT configures it rather than reinventing it. Userspace stays small, auditable, and fast to boot.

Quick Start

Build the userspace binary, bake it into the XIP kernel image, and boot it in QEMU.

Prerequisites

  • Rust nightly with rust-src (pinned by rust-toolchain.toml).
  • An OpenWrt musl cross toolchain for mipsel_24kc — it provides the musl gcc that drives the final link (the tier-3 target ships no self-contained CRT).
  • For the kernel image: clang, a mipsel-linux-gnu- binutils, cpio, qemu-system-mipsel.

Build the userspace binary

bash 
scripts/build.sh
# -> target/mipsel-unknown-linux-musl/release/lwrt

build.sh auto-detects your OpenWrt staging_dir, wires the linker and STAGING_DIR through the environment (so .cargo/config.toml stays machine-neutral), regenerates vendor/libunwind.a, and produces the stripped lwrt binary. Override paths with LWRT_OPENWRT, LWRT_TOOLCHAIN, LWRT_TARGET.

Bake and boot the XIP image

bash 
# point at your kernel tree; result is <kernel>/build/out/xip-bios.bin
LWRT_KERNEL=/path/to/mips-xip-kernel scripts/mkimage.sh

qemu-system-mipsel -M malta -cpu 24Kf -m 256 \
  -bios xip-bios.bin -nographic
Expected output. On a successful boot the console prints LWRT init: PID 1 starting.

The Multicall Binary

Like busybox, LWRT decides which applet to run from how it was invoked. It prefers the basename of argv[0] (so symlinks like /sbin/dhcp → lwrt work), and falls back to the first argument when called as the bare lwrt binary.

rust 
// applet selection: argv[0] basename, else argv[1]
let arg0 = args.first()
    .map(|s| s.rsplit('/').next().unwrap_or(s))
    .unwrap_or("lwrt");

let (applet, rest) = match arg0 {
    "lwrt" | "" => match args.get(1) {
        Some(a) => (a.as_str(), &args[2..]),
        None => ("help", &[][..]),
    },
    other => (other, &args[1..]),
};

let code = match applet {
    "init"  => init::run(rest),
    "ifup"  => applet_ifup(rest),
    "dhcp"  => dhcp::run(rest),
    "dns"   => dns::run(rest),
    "fw"    => fw::run(rest),
    "wg"    => wg::run(rest),
    "httpd" => httpd::run(rest),
    "ipkg"  => ipkg::run(rest),
    _ => { usage(); 1 }
};

One module per applet, one run(&[String]) -> i32 entry point each. Adding a feature is adding a module and a match arm — no new process, no new binary.

init — PID 1

The init applet is a real PID 1. It mounts the pseudo-filesystems the rest of the system expects, brings up networking, then supervises the long-lived services and reaps orphaned children.

  • Mounts /proc, /sys, /dev (devtmpfs), /tmp — failures surface as clear console errors that map directly to a missing kernel config.
  • Network bring-up from the config: LAN address, WAN client, default route.
  • Supervision: restarts the configured services (dns, dhcp, httpd …) if they exit, and reaps zombies via waitpid.
bash 
lwrt init        # normally invoked by the kernel as /sbin/init

net / ifup

Interface configuration through ioctl and rtnetlink — assign an address and netmask, optionally install a default route. This is the manual counterpart to what init does automatically from the config.

bash 
ifup <name> <ip> <netmask> [gateway]
# e.g.
ifup br-lan 192.168.1.1 255.255.255.0
ifup eth0.2 10.0.0.5 255.255.255.0 10.0.0.1

dhcp

A combined DHCP server (for the LAN) and client (for the WAN), replacing dnsmasq's DHCP role and OpenWrt's WAN udhcpc. Packets are parsed with a shared, allocation-light option walker that iterates the TLV stream and stops cleanly on truncation — a malformed packet never panics a supervised service.

bash 
dhcp server      # lease out the LAN pool
dhcp client      # acquire a WAN lease, set address + route
Robust by construction. Option parsing is a single Iterator over the packet body; address fields parse through typed helpers that return Option, so bad input is dropped rather than unwrapped.

dns

A forwarding resolver with a small cache, plus resolution of local DHCP-lease names — dnsmasq's DNS role in miniature. Queries it can't answer locally are forwarded to the configured upstream; answers are cached by TTL.

bash 
dns              # bind :53, forward + cache

fw — nftables loader

The firewall applet programs an nf_tables ruleset over NETLINK_NETFILTER: a table, base chains hooked into forward/postrouting, and the NAT + filter rules. There is no userspace packet path — the applet just builds the netlink batch and hands it to the kernel's nftables VM.

The whole ruleset is encoded into a single netlink batch buffer (NFNL_MSG_BATCH_BEGIN … rules … NFNL_MSG_BATCH_END) and sent in one shot; the applet then drains the kernel's ACKs.

bash 
fw               # load the NAT + filter ruleset (table "lwrt")
Platform note. On the QEMU malta XIP target the netfilter stack is omitted from the kernel because the -bios window leaves too little room below the R_MIPS_26 boundary. The real MT7628 flash window has 64 MB of headroom and carries the full firewall. See Porting.

wg — WireGuard front end

WireGuard embodied: the crypto, the transport and the routing all live in the kernel module; this applet is the thin tool that creates the wg0 interface (rtnetlink) and pushes the key, port and peers into it (WG_CMD_SET_DEVICE over genetlink). No userspace crypto.

1
Create the link

rtnetlink RTM_NEWLINK with kind wireguard (EEXIST is treated as success).

2
Resolve the family

genetlink CTRL_CMD_GETFAMILY to find the WireGuard family id.

3
Set the device

WG_CMD_SET_DEVICE: private key, listen port, and each peer (public key, endpoint, allowed-IPs, keepalive).

4
Address & up

Assign the tunnel address and bring the interface up.

bash 
wg up            # create wg0, push key/port/peers, bring up
wg down          # tear the link down

Because WireGuard uses cryptokey routing — a packet is only accepted if it arrives from the peer whose allowed-IPs cover its source — much of the firewalling a classic VPN needs is subsumed by the tunnel itself, keeping the fw ruleset lean.

httpd — admin web UI

A compact embedded web server that serves a single-page admin UI and a small JSON API — the role LuCI + uhttpd play in OpenWrt, in a fraction of the footprint. The UI reads and writes the same INI-style config the rest of LWRT uses.

bash 
httpd            # serve the admin SPA + JSON API
Hardening in flight. Authentication, CSRF protection, output escaping and secret redaction are tracked work, not yet complete — do not expose the UI to an untrusted network. See bugs.md.

ipkg — package manager

A minimal package manager: it fetches an index, lists and installs .ipk archives (ustar payloads) and removes them — opkg's core flow without the C dependency tree.

bash 
ipkg update
ipkg list
ipkg install <name>
ipkg remove  <name>
Roadmap. SHA-256 integrity verification, size limits, upgrade and refcount handling are planned (see the parity map and incomplete.md).

Internals: the Netlink Builder

Both the firewall (NETLINK_NETFILTER / nf_tables) and WireGuard (NETLINK_GENERIC / genetlink) speak attribute-encoded netlink. LWRT encodes every message through NlBuf — a single growable buffer that appends each header and attribute in place and backpatches the length fields of nested attributes and messages when their scope closes.

The naive approach — returning a fresh Vec<u8> per attribute and extend-ing them together — allocates dozens of short-lived buffers to emit one ruleset. NlBuf allocates once and grows.

Scopes that can't be left open

A nested attribute or a message is opened with a begin_* call that returns a Scope marker and writes a placeholder length; the matching end_* backpatches the real length. The marker is #[must_use], so forgetting to close a scope (which would leave a zero length on the wire) is a compile-time warning.

rust 
let mut b = NlBuf::new();
let m = b.begin_message(family, NLM_F_REQUEST | NLM_F_ACK, 1);
genlmsghdr(&mut b, WG_CMD_SET_DEVICE, 1);
b.attr_str(WGDEVICE_A_IFNAME, ifname);
b.attr_u32_le(WGDEVICE_A_FLAGS, WGDEVICE_F_REPLACE_PEERS);
b.attr(WGDEVICE_A_PRIVATE_KEY, privk);
b.attr_u16_ne(WGDEVICE_A_LISTEN_PORT, port);

let peers = b.begin_nested(WGDEVICE_A_PEERS);   // Scope (must be closed)
let peer  = b.begin_nested(peer_idx);
b.attr(WGPEER_A_PUBLIC_KEY, &pubk);
b.end_nested(peer);
b.end_nested(peers);
b.end_message(m);

nl::send(fd, b.as_slice())?;

Wire format reference

ElementEncoding
Attribute[len u16 le = 4+payload][type u16 le][payload][pad to 4] — len excludes the pad.
Nested attrAn attribute whose type carries the NLA_F_NESTED (0x8000) flag; its payload is a sub-stream of attributes.
Message[nlmsg_len u32 le = 16+body][type u16][flags u16][seq u32][pid u32 = 0][body][pad to 4]
Verified by tests. Unit tests assert 4-byte alignment, correct length fields, the nested flag, and that for_each_attr reads back exactly what was written — so the builder is byte-identical to the hand-rolled encoding it replaced.

Configuration

LWRT reads one INI-style file — /etc/lwrt/config — with [section] headers and key = value lines. Every applet pulls its settings from the same parsed map; there is no UCI, no per-daemon format.

ini 
[lan]
ifname = br-lan
address = 192.168.1.1
netmask = 255.255.255.0

[wan]
ifname = eth0.2
proto = dhcp

[wireguard]
enabled = 1
private_key = <base64 32-byte key>
listen_port = 51820
address = 10.0.0.1/24

[peer0]
public_key = <base64 32-byte key>
endpoint = 203.0.113.7:51820
allowed_ips = 10.0.0.2/32, 10.0.0.0/24
keepalive = 25

Any section whose name begins with peer and carries a valid public_key becomes a WireGuard peer. Missing or malformed values are skipped gracefully rather than aborting the applet.

Building

LWRT targets the tier-3 mipsel-unknown-linux-musl triple, built with nightly and -Zbuild-std, optimised hard for size.

opt-levelzoptimise for size, not speed
lto / codegen-unitstrue / 1cross-module inlining + dead-code elimination
striptruedrop symbols from the shipped binary
build-stdstd, panic_abortrebuild std for the tier-3 target

The final link is driven by the OpenWrt musl gcc (the tier-3 target ships no self-contained CRT), with vendor/libunwind.a as a copy of libgcc_eh.a. scripts/build.sh wires all of this through the environment so .cargo/config.toml stays machine-neutral.

Porting to Another Router

LWRT itself is board-agnostic — it talks to the kernel through standard ioctls and netlink. Porting is really kernel porting.

  1. Pick the flash XIP window. Set CONFIG_XIP_PHYS_ADDR to the SoC's memory-mapped SPI/NOR flash base. On MT7628 that is 0x1c000000 (KSEG0 0x9c000000).
  2. Mind the R_MIPS_26 boundary. MIPS jal can only jump within a 256 MB-aligned segment. Keep the kernel's flash-resident text below the next 0x_0000000 boundary above the window. The malta -bios window at 0x1fc01000 leaves only ~4 MB before 0xa0000000 — which is why the malta defconfig omits the netfilter stack; the MT7628 window at 0x9c000000 has 64 MB of headroom and fits the full firewall.
  3. Write a defconfig for the board (NIC driver, DSA/switch, WiFi) modelled on configs/lwrt_qemu_malta_defconfig.
  4. Adjust rootfs/etc/lwrt/config — interface names (lan.ifname, wan.ifname), addresses, radios.
Two malta ceilings. The malta -bios path both limits usable flash text to ~4 MB and byteswaps only the first ~3.9 MB of the image on mipsel. Netfilter and bridge support are therefore deferred to the MT7628 target, which has neither limitation.

Debugging on Real Hardware

  • Serial console early. Keep earlycon=ttyS0 (or the board's UART) on the kernel command line. Rust init's kernel-ABI mismatches then surface as syscall errors before anything else is up — e.g. mount /proc: No such device means CONFIG_PROC_FS is off; Function not implemented means a kernel subsystem is missing.
  • Netlink, not strace. There is no strace on the device. To debug firewall application, read the kernel's netlink/netfilter logs — the fw applet programs nftables over NETLINK_NETFILTER and the kernel reports rejections.
  • Binary integrity. Make sure the musl the toolchain links against is compatible with the kernel you boot; a mismatch shows up as ENOSYS/EINVAL from otherwise-correct code.

Glossary

XIP (Execute-In-Place)
Running kernel code directly from flash without copying it to RAM — possible on MIPS because KSEG0/KSEG1 are unmapped segments.
Multicall binary
A single executable that behaves as many tools, selected by the name it is invoked under (busybox-style).
netlink
The kernel↔userspace socket protocol used to configure networking, netfilter and WireGuard. Messages are streams of type-length-value attributes.
genetlink
Generic netlink — a multiplexing layer over netlink; WireGuard registers a family resolved at runtime by name.
nf_tables
The kernel's in-kernel packet-classification VM that replaced iptables; LWRT's fw applet compiles a ruleset into nftables netlink batches.
cryptokey routing
WireGuard's model where a peer's allowed-IPs both select the encryption key for outbound packets and authorise inbound source addresses.
R_MIPS_26
The MIPS relocation for the jal instruction, which can only target an address within the same 256 MB-aligned region — a hard ceiling on flash-resident kernel text.

FAQ
